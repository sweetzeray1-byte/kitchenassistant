Goal: rank the Delisio web ([[web-is-separate-nextjs]]) higher in search, driven by a keyword CSV + a strategy report (benchmarking Allrecipes). Two keyword pillars to capture: **ground beef/hamburger** (500k/mo terms, Low/Med competition) and **lunch/work meals** (several +900% YoY surging long-tails).

**Audit (as of 2026-06-09):** Recipe pages are already strong — server-rendered `generateMetadata` + full `Recipe` JSON-LD (ingredients, HowToStep, nutrition, ISO-8601 times) in `web/src/app/recipe/[id]/page.tsx`; ISR on home/recipe; `next/image` + Supabase remote patterns. **Gaps:** no `sitemap.ts`, no `robots.ts`, no `metadataBase`, no canonical tags, no `ItemList`/`BreadcrumbList` JSON-LD, `discover` is fully client-side (not indexable), no programmatic taxonomy hubs, recipes are UUID `/recipe/[id]` (no slugs), no `aggregateRating` (no ratings system exists).

**Plan (user chose "full plan first" then "start with Phase 1 and 2" on 2026-06-09):**
- Phase 1 (DONE 2026-06-09): `web/src/lib/seo.ts` (SITE_URL from `NEXT_PUBLIC_SITE_URL`, fallback `kitchenassistant-pi.vercel.app`; ItemList/Breadcrumb/jsonLdScript helpers); metadataBase + title template + keywords in layout; `app/robots.ts`; home + recipe canonical; ItemList on home/discover; BreadcrumbList on recipe page.
- Phase 2 (DONE 2026-06-09): backend `getRecipesForSitemap()` + `GET /api/recipes/sitemap` (public recipes only, see [[delisio-public-recipes-filter]], capped at range 0-9999 — paginate if >1000 public recipes); `web/src/app/sitemap.ts` (v16 `generateSitemaps`, `id` is a Promise — NOT the old number signature in the PDF); discover refactored to server component (`app/discover/page.tsx` SSR-fetches initial grid + canonical) feeding `components/discover-client.tsx` via react-query initialData. Web build + backend tsc both green.
- **ACTION for user:** set `NEXT_PUBLIC_SITE_URL` in Vercel to the real domain (else canonical/sitemap/OG use the vercel.app fallback). Submit `/sitemap.xml` in Google Search Console after deploy.
- Phase 3 v1 (DONE 2026-06-09): data-driven hubs. `web/src/lib/taxonomy.ts` = pure config of HubDefs (slug, h1, metaTitle/Desc, pre-written unique intro, discover filter, related, generateSeed). Routes `app/ingredients/[ingredient]/page.tsx` + `app/meals/[meal]/page.tsx` (server, generateStaticParams, generateMetadata+canonical, ItemList+Breadcrumb JSON-LD, notFound for unknown slugs). `web/src/lib/hub-data.ts` fetches via existing `/api/recipes/discover`. Shared UI `components/hub-page.tsx` (intro, grid, AI-generate CTA when sparse → /discover?q=, related links). Hubs added to sitemap + a footer "Browse" column. Seed hubs: ingredients ground-beef/hamburger/chicken (filter by title `query`, no per-ingredient category exists), meals lunch/dinner/breakfast/dessert (filter by real `category`). Build prerenders all 7 as SSG w/ 10m ISR. Decisions taken: hub-slugs only (recipes stay /recipe/[id]); pre-written intros; ingredient filter = title ILIKE for v1.
- Phase 3 v1.1 (DONE 2026-06-09): (a) BACKEND ingredient search — `getRecipesByIngredient(term)` in supabaseService.ts matches title + any `ingredients[]` element (case-insensitive substring) by fetching public recipes (cap range 0-999) and filtering in JS (no DB migration; ISR-cached). Added shared `mapRecipeRow` helper. Wired via new `ingredient` query param on GET /api/recipes/discover (branches to getRecipesByIngredient). SCALE-UP: replace JS filter with Postgres RPC `EXISTS(SELECT 1 FROM unnest(ingredients) i WHERE i ILIKE '%term%')` + GIN index when catalog grows. (b) EXPANDED taxonomy to 16 hubs: added 3rd hub kind "category" → route `app/recipes/[category]/page.tsx` mapping to real categories (healthy, quick-easy, vegetarian, vegan, gluten-free, pasta, soup, salad, seafood). Ingredient hubs now use `ingredient` filter (ground-beef, chicken); hamburger stays title `query` to stay distinct from ground-beef (avoid near-dupe). hubPath/getHub handle 3 kinds; sitemap + footer auto-include via ALL_HUBS. All 16 prerendered SSG/10m ISR; build + backend tsc green.
- Phase 3 NEXT: deeper intersections (/meals/lunch/work, /recipes/[...slug] long-tail) need audience/modifier filters that don't exist; AI-generate intros for scaling beyond hand-written hubs; more CSV terms. Backend categories are a FIXED set of 21 ids in src/config/categories.ts; tags are free-form (<=3).
- Phase 4: aggregateRating (needs new ratings system), multi-aspect-ratio schema images, OG image generation, recipe slug migration.

**Open questions blocking Phases 2–3:** (1) production canonical domain? (2) hub-slugs-only vs. full recipe-slug migration? (3) pre-write top ~30 hub intros vs. AI-generate all?
